## Plan Review: Step 4: Update frontend identity handoff

### Summary
The four outcome-level checkboxes correctly target the key frontend identity seams that need to drop `ibkrConid` from canonical handoff: provisional key construction, module navigation/query helpers, route query-param emission, and the authority model between backend-returned keys and optimistic client state. No missing outcomes were identified.

### Issues Found
*None*

### Suggestions
- **Watchlist `ibkrConid` type preservation:** The `WatchlistSymbol` type and `TerminalMarketMonitorBaseRow` type both carry an `ibkrConid` field today. When executing checkbox 3 ("removed or isolated"), consider keeping `ibkrConid` on deserialized response types (so `WatchlistSymbol` can still hold it as provider-specific metadata the backend may return) but removing or zeroing it from the `NormalizedInstrumentIdentity` canonical type and from `createProvisionalInstrumentKey` output. The `toNavigationInstrumentIdentity` helper in `terminalMarketMonitorWorkflow.ts` is a conduit that should stop forwarding it to navigation intents.
- **Route backward compatibility:** `terminalRoutes.ts` `createTerminalRouteIdentity` currently parses `ibkrConid` from query params. Since the backend API still accepts it as a legacy alias (Step 3), the frontend should keep parsing it for backward compat but immediately normalize it into `provider`/`providerSymbolId` before the identity tuple crosses module seams. The regression suite tests added in Step 1 already assert `ibkrConid` is NOT read from query params via `firstTerminalQueryValue(searchParams.ibkrConid)` — the worker should decide whether to remove the parse line entirely or keep it with an explicit normalization comment.
- **Test updates:** Several frontend apphost tests (`frontend-chart-watchlist-default-tests.sh`, `frontend-stock-chart-visibility-tests.sh`, `frontend-terminal-chart-analysis-tests.sh`, `frontend-terminal-market-monitor-tests.sh`, `frontend-terminal-route-architecture-tests.sh`) currently assert the *presence* of `ibkrConid` in the frontend source (e.g., `assert_file_contains "$identity" "params.set('ibkrConid', …)"`). These will break after the implementation changes. The Step 6 (Testing & Verification) checklists handle "run affected frontend/apphost identity contract tests" and "fix all failures" — the worker should anticipate these test updates as a natural consequence of the Step 4 code changes.
- **`parseIbkrConid` helper fate:** Once `ibkrConid` is removed from canonical frontend identity types and key construction, the `parseIbkrConid` function in `instrumentIdentity.ts` may only be needed internally when converting IBKR provider data to the `providerSymbolId` field. Consider whether it can be made private/module-local or removed if all callers already obtain `providerSymbolId` from provider data directly. The function is used in `getSearchResultIdentity`, `getTrendingSymbolIdentity`, and `getMarketDataIdentity` — each of which could instead derive `providerSymbolId` from the underlying response data.
